# kaoshi UI

先看页面结构，再改样式。UI 问题通常是信息架构、组件语义、状态流和布局约束的问题。

## UI 铁律

- 默认使用 Element Plus、Vue Router 和成熟考试平台已验证的信息架构与交互模式。
- 禁止自造基础 UI 轮子；表格、分页、表单、上传、选择器、权限多选、弹窗、抽屉、步骤条、标签、答题卡、计时器优先使用成熟组件或成熟模式封装。
- 修改 UI 前必须对照 `yfexam-exam` 等成熟参考项目的页面结构、控件语义、状态流和布局约束；不能凭个人偏好硬造页面体验。
- 只有当前业务边界、类型契约和验证方式证明成熟组件无法满足时，才允许封装 kaoshi 自有组件。

## 必读上下文

修改 UI 前先读取目标页面、目标组件、路由配置、API client、状态管理、全局样式、`spec.md` 中的当前产品事实。

## 信息架构

- 管理端服务高频维护：列表、筛选、批量操作、表单、详情、审核和结果查看。
- 考试端服务稳定作答：考试信息、倒计时、题目区、答题卡、保存状态、提交确认和结果反馈。
- 题库、试题、试卷、考试、成绩必须有清晰路径。
- 列表页负责检索、筛选和进入编辑。
- 编辑页负责字段填写、校验和保存。
- 详情页负责只读查看和审计信息。
- 角色权限配置必须用复选框、分组、全选和清晰的已选状态；禁止把大量权限塞进一个下拉框让用户猜。

## 组件语义

- 按钮只用于真实命令。
- 表格用于可比较数据。
- 表单用于编辑。
- Dialog 用于确认和短流程。
- Drawer 用于辅助详情。
- Tabs 用于同一对象的不同视图。
- Steps 用于有顺序的考试发布或组卷流程。
- Checkbox/CheckboxGroup 用于权限、菜单等多选配置。

## 布局规则

- 管理端保持密度适中、可扫描、可批量操作。
- 考试端保持稳定、克制、无干扰。
- 长文本必须换行或省略。
- 表格列必须明确宽度、固定列和溢出策略。
- 表单项必须有清楚 label、校验提示和错误状态。
- 页面不使用装饰性大面积渐变和空洞宣传卡片。
- 不在页面上写“这个功能是什么/如何使用”的说明文案；用清晰的字段名、控件和状态表达功能。

## 验证

收尾前检查文本不溢出、按钮不被挤压、表单错误提示可见、空状态/加载态/失败态存在、移动宽度下考试端仍可作答。
