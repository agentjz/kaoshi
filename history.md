## 一句话

kaoshi 不是一个 CRUD 后台，也不是一个只给人看的 demo。它是一套围绕考试事实运转的平台：建题库、组试卷、发布考试、考生作答、保存作答事实、自动评分、人工阅卷、成绩归档、结果复盘。

这个项目真正的难点不是“页面多不多”，而是考试事实不能乱：发布后题库怎么改，都不能污染已经发布的试卷；考生提交后管理端怎么改，都不能污染已经提交的作答；成绩必须能从当时的发布快照和作答快照复现出来。

这个项目一开始不是先堆业务页面，而是先建立协作方式。owner 已经在多个项目里反复验证过：如果没有 `AGENTS.md`、`spec.md`、`plan.md` 和项目级 skill，AI 很容易只按一句话局部改，结果越改越乱。

走过的弯路：

- 快速演进项目不能被历史包袱拖住；当前生产级目标高于旧实现。

## 阶段 01：从骨架变成全栈考试平台

时间：2026-07-01

关键提交：

- `a6ddd18 Initialize kaoshi exam platform`
- `93b1012 Polish project README`

当时要解决的问题：

项目需要从文档骨架变成一个能运行、能登录、能管理、能考试的全栈系统。这个阶段的目标不是只做 UI，也不是只做后端接口，而是让后端、前端、数据库、测试、脚本同时成形。

当时做了什么：

- 后端建立 Java 21、Spring Boot 3、MyBatis-Plus、Spring Security、JWT 主体。
- 数据库建立 MySQL / Redis / Flyway 初始化脚本。
- 前端建立 Vue 3、TypeScript、Vite、Element Plus 主体。
- 建立管理员登录、当前用户、角色、权限、菜单、用户管理。
- 建立题库、试题、试卷、考试、作答、成绩的初版模型。
- 建立 Docker Compose 和本地启动/测试脚本。
- 建立 JUnit、MockMvc、Vitest、Playwright 验证入口。

当时走对的地方：

- 一开始就不是单页 demo，而是完整工程。
- 一开始就把用户、权限、题库、考试、成绩放到同一个业务主线里。
- 一开始就有后端测试、前端测试和真实浏览器验收的意识。

当时的问题：

- 初版里 `paper` 作为独立试卷模块存在，后来发现它和 `exam` 的生命周期边界会冲突。
- 初版页面和接口很多，但“考试事实如何长期不变”还没有完全成为核心。
- 初版更像通用考试后台，还没有明确 CET-4 的产品表达。

最后留下来的实践：

- 技术栈保留下来。
- 全栈工程结构保留下来。
- paper 独立模块后来被删除，考试主线收敛到 `exam`。

## 阶段 02：组织、导入导出、文件和真实验收

时间：2026-07-02

关键提交：

- `ae5bc55 feat: add department management (infinite tree), Excel import/export, and file upload system`

当时要解决的问题：

考试平台不能只有题目和考试。真实组织里需要部门、用户导入导出、文件附件、题目媒体、真实浏览器验收。否则它只能当演示，不能当平台。

当时做了什么：

- 增加部门无限树。
- 增加用户 Excel 导入导出。
- 增加通用文件上传。
- 增加音频、图片、文件附件。
- 增加考试准备页、考试会话页、成绩详情页。
- 增加多个 Playwright 真实链路：系统管理、考试管理、在线考试。
- 增加 `start_dev.py`、`start_test.py`、`start_browser_test.py` 等入口。

走过的弯路：

- 早期管理端菜单较多，权限、菜单、试卷、题目等页面都有显性入口，容易变成普通后台系统。
- 文件和附件一开始只是功能点，后来才变成考试材料、题目附件、发布快照的重要事实。

最后证明正确的实践：

- 部门、Excel、文件不是锦上添花，而是生产平台必须能力。
- 真实浏览器验收必须保留，因为很多问题单元测试看不出来。
- 文件附件必须参与考试快照和作答展示，否则编辑后的素材会污染历史考试。

## 阶段 03：考试生命周期重建

时间：2026-07-02

关键提交：

- `9e95609 重建考试生命周期与项目规则`

当时要解决的问题：

初版 `paper` 和 `exam` 的边界不清。试卷、考试、作答、成绩如果分散成多个主线，后续发布、撤销、删除、作答、评分都会变复杂。

当时做了什么：

- 删除独立 `paper` 模块。
- 把考试保存、发布、作答、评分收敛到考试生命周期。
- 增加首次登录强制改密。
- 增加考试发布、保存草稿、作答、提交、评分相关边界。
- 更新项目规则：当前事实高于历史形态，旧路径可以删除重建。

走过的弯路：

- 试图把 paper 当成独立实体，这对简单系统看起来合理，但对当前平台不合理。
- 如果 paper 和 exam 同时存在，发布快照到底来自 paper 还是 exam 会变得含糊。

最后证明正确的实践：

- 考试是主线。
- 试卷只是考试发布前后的事实载体，不应脱离考试生命周期成为另一个主系统。
- 删除旧模块比兼容旧错误更清楚。

## 阶段 04：管理端信息架构和题库管理

时间：2026-07-03

关键提交：

- `3699feb 规划管理端信息架构重建`
- `452fa23 标记管理端重构计划启动`
- `39868c7 题库管理模块`
- `604dcd7 完善管理端体验和成绩链路`

当时要解决的问题：

系统已经有功能，但管理端不够像一个人能长期使用的平台。菜单太散，页面像接口集合，题库和试题关系不够清楚，成绩链路也不够顺。

当时做了什么：

- 规划管理端信息架构。
- 系统管理入口收束为角色、部门、用户。
- 考试管理入口收束为题库管理、考试管理。
- 题库管理变成左侧题库树、右侧试题列表的工作台。
- 完善题库分类、题库、试题列表、试题筛选。
- 完善成绩详情、管理员按考试查看成绩、写作阅卷路径。

走过的弯路：

- 早期显性入口太多，菜单像数据库表。
- 权限、菜单这类底层数据如果直接暴露给普通用户，会让产品显得像开发者后台。
- 题库和试题如果拆成两个割裂页面，用户会不知道当前题属于哪个题库。

最后证明正确的实践：

- 菜单应该按用户任务组织，不按数据库表组织。
- 权限和菜单可以作为角色配置的数据来源，但不必成为普通显性入口。
- 题库树 + 当前题库试题列表，是比散页面更自然的管理心智。

## 阶段 05：快照成为核心

时间：2026-07-03 到 2026-07-04

关键提交：

- `5c0c42c 完善考试作答保存与组卷排序`
- `60aeefe 完善考试生命周期和写作题阅卷闭环`
- `dd1de6d feat: exam workflow, question import, and admin UI overhaul`

当时要解决的问题：

考试平台真正难的是历史事实。题库会改，题目会改，选项会改，附件会改，但已经发布的考试和已经提交的作答不能跟着变。否则成绩复盘就不可信。

当时做了什么：

- 拆出 `ExamAttemptWorkflow`、`ExamGradingWorkflow`、`ExamPaperWorkflow`、`ExamResponseAssembler` 等职责。
- 加强考试草稿保存、读回、人工排序、分值配置。
- 发布时生成发布快照。
- 开始作答时生成 attempt 快照。
- 保存整卷答案快照。
- 提交后锁定。
- 选择题自动评分。
- 写作题进入人工阅卷。
- 阅卷可以保存进度，最后显式完成。
- 成绩详情展示考生答案、正确答案、解析和附件。

走过的弯路：

- 只保存题目 ID 不够。题库后续编辑会污染已发布考试。
- 只保存最后答案不够。刷新、重进、提交锁定、成绩复盘都需要完整作答快照。
- 阅卷不能静默覆盖最终成绩，必须有进度和完成边界。

最后证明正确的实践：

- 发布快照和作答快照是考试平台的底层事实边界。
- 评分必须基于快照，而不是当前题库。
- 管理端编辑能力不能破坏已发布、已作答、已评分的历史事实。

## 阶段 07：复杂题组路线被删除

时间：2026-07-04 到 2026-07-05

关键提交：

- `457f99f feat: 重构题库内容树和考试快照`
- `25d5ffc 重建答题卡试卷模式`

当时要解决的问题：

四级考试有听力、阅读、写作、材料和题号关系。项目一度尝试用复杂题组、题目节点、题库包、共享选项、JSON 题组导入来表达这些结构。

当时做过的路线：

- `QuestionContentController`
- `QuestionContentService`
- `QuestionContentMapper`
- `QuestionNode`
- `QuestionNodeOption`
- `QuestionNodeAttachment`
- `QuestionBankPackageService`
- `QuestionBankPackageWorkbookCodec`
- `QuestionBankPackageZipCodec`
- `QuestionSetResourceImporter`
- `question-sets/*.json`
- 题组树、共享选项、题库包导入导出

这条路为什么看起来合理：

四级真题确实有材料、题组、共享上下文、音频、阅读材料、题号等复杂结构。如果直接从试题角度建模，会觉得应该把这些都做成题库内容树。

后来为什么证明不对：

- 它把命题材料、题库内容、考试试卷、答题卡混成了一套结构。
- 管理端会变成开发者式编辑器，而不是考务人员能理解的考试配置。
- 复杂题组结构让导入导出、发布快照、作答快照、成绩复盘全都变重。
- 对当前产品主线来说，题库只需要维护基础试题；答题卡试卷可以把材料和答案卡作为考试的一部分处理。

最后做了什么：

- 删除 question content tree。
- 删除 question package。
- 删除复杂题组 JSON 资源。
- 删除共享选项和题目节点路线。
- 题型收敛为 `SINGLE_CHOICE`、`MULTIPLE_CHOICE`、`WRITING`。
- 把四级材料和答题卡从“题库复杂结构”转为“考试答题卡模式”。

最后证明正确的实践：

复杂不是错，但复杂必须落在正确边界。四级真题材料和答题卡更像一次考试的试卷事实，不应该塞进题库基础试题模型里。

## 阶段 08：答题卡试卷模式

时间：2026-07-05

关键提交：

- `25d5ffc 重建答题卡试卷模式`
- `db3d93c 重建答题卡试卷生产链路`

当时要解决的问题：

四级考试的真实心智不是每道题都有完整题干卡片。很多时候学生看到的是一份材料或试卷，然后在答题卡上填 1、2、3、4 题答案。系统需要支持“试卷材料”和“答题卡答案”分离。

当时做了什么：

- 增加考试模式：
  - `STRUCTURED`：结构化试卷，从题库题目组卷。
  - `ANSWER_SHEET`：答题卡试卷。
- 增加试卷材料分组。
- 增加试卷材料文件。
- 增加答题卡条目。
- 管理端可以维护材料、附件、外部链接、排序、题号、题型、选项范围、正确答案、分值。
- 发布答题卡试卷时生成材料发布快照。
- 作答页支持上方材料、下方答题卡、右侧题号导航。
- 答题卡选择题自动评分。
- 答题卡写作题进入人工阅卷。
- demo 也支持答题卡试卷体验。
- 拆分 `ExamAnswerSheetWorkflow`、`ExamMaterialWorkflow`、`ExamStructuredPaperWorkflow`。
- 前端拆出 `ExamAnswerCardEditor.vue`、`ExamMaterialEditor.vue`、`ExamMaterialViewer.vue`、`useExamEditor.ts` 等模块。

走过的弯路：

- 只做 URL 材料不够。用户需要上传 PDF、图片、音频、文件，也需要从已有附件选。
- 只把答题卡当 UI 不够。它必须进入保存、读回、发布快照、作答快照、评分和阅卷。
- 总分和题量计算如果只看结构化试卷题目，会漏掉答题卡条目。
- 答题卡试卷不能重新引入复杂题组结构，否则前一阶段删除就失去意义。

最后证明正确的实践：

- 答题卡是一种考试模式，不是题库题型。
- 材料属于试卷事实，发布后必须快照。
- 答案卡条目属于试卷事实，提交后必须可评分、可复盘。
- 学生端 UI 要贴近真实作答心智：看材料，填答题卡。

## 业务主线演进

这个项目的业务目标从一开始就不是炫技，而是实用。

早期它像一个现代考试平台：有后台、有用户、有题库、有考试、有成绩。随后它逐步被拉回真实考试业务：考试不能只是表单，题库不能只是 CRUD，成绩不能只是一个分数，作答不能只是一个提交按钮。

最后形成的业务主线是：

1. 管理员维护组织、用户、角色。
2. 命题人维护题库和试题。
3. 考务管理员创建考试草稿。
4. 考试发布时固化发布快照。
5. 考生进入考试，系统生成作答快照。
6. 考生保存、刷新、继续、提交。
7. 系统基于快照自动评分选择题。
8. 写作题进入人工阅卷。
9. 成绩锁定并可复盘。

这条主线里最重要的词是“固化”。没有固化，平台就只是当前数据库状态的展示；有固化，平台才开始像考试系统。

## 协作史和沟通教训

### 教训 01：只写技术栈不够

这个项目当然有技术栈：Java、Spring Boot、MySQL、Redis、Vue、TypeScript、Element Plus、Playwright。但如果历史文档只写技术栈，它就没有说到项目的本质。

真正的本质是业务事实演进：为什么 paper 被删，为什么复杂题组被删，为什么发布快照成为核心，为什么答题卡属于考试模式，为什么当前又要补身份入口。

后续 AI 接手时，不要把 kaoshi 理解成“一个 Spring + Vue 项目”。它首先是一个考试平台。

### 教训 02：不要把页面数量当成熟度

早期系统有很多页面：权限、菜单、试卷、题目、题库、考试、成绩。页面多不代表成熟。成熟的表现是用户任务清楚：系统管理看角色、部门、用户；考试管理看题库管理和考试管理；考试端看考试中心、作答、成绩。

后续新增页面时必须问：这是用户任务，还是数据库表暴露？

### 教训 03：不要保留错误旧路径

paper 模块、复杂题组、题库包、question content tree 都被删过。删除不是失败，是收束。

kaoshi 的规则是当前生产级目标高于历史形态。旧路径如果不服务当前主干，就删掉。不要写兼容，不要写历史残留，不要让测试和文档继续出现已经没有意义的旧概念。

### 教训 04：测试要保护真实产品行为

这个项目很早就有真实浏览器验收。它不是形式主义，因为考试平台的很多问题只有真实链路能发现：登录后跳转、菜单顺序、题库选择、附件展示、作答保存、刷新读回、提交锁定、阅卷读回。

后续不要只补 mock。单元测试保护规则，浏览器验收保护人真的怎么用。

### 教训 05：demo 不能变成第二套系统

GitHub Pages demo 是对外展示入口，但它不能和真实系统分叉。当前正确做法是：同页面、同路由、同类型，demo 只是内存 adapter。

后续加注册、找回密码、答题卡、成绩发布策略时，demo 也应该沿用同一套用户心智，而不是复制一套假页面。

## 已删除或被否定的路线

### 独立 paper 模块

结论：删除。

原因：试卷不能脱离考试生命周期。发布、撤销、作答、评分、成绩复盘都需要围绕考试主线。

### 权限和菜单作为普通显性入口

结论：收束。

原因：权限和菜单是角色配置的数据来源，不应该让普通管理端像数据库后台。

### 复杂题组 / question content tree / question package

结论：删除。

原因：它把题库、材料、试卷和答题卡混在一起。当前主线用答题卡试卷模式承载四级材料和答案卡，更清楚。

### 只保存题目 ID 的考试

结论：否定。

原因：题库编辑会污染已发布考试。必须有发布快照。

### 只保存最后答案的作答

结论：否定。

原因：刷新读回、提交锁定、成绩复盘都需要作答快照。

### demo 单独写一套页面

结论：否定。

原因：会分叉，后续真实能力和演示能力会互相背离。

## 最后证明正确的路线

### 当前事实高于历史形态

代码能编译不够。考试平台必须用真实后端、真实前端、真实数据库、真实浏览器跑主路径。
