# kaoshi 题库分类管理补齐计划

## 需求

- 题库工作台必须表达真实业务层级：试题分类 -> 题库 -> 试题。
- 当前只能在初始化的“默认试题分类”下新建题库，缺少新建分类、编辑分类和删除空分类能力，需要补齐完整生命周期。
- 管理端只保留“题库管理”显性入口，不恢复独立“试题管理”模块。
- kaoshi 是快速演进项目。本次按当前生产级主干直接补齐接口、页面、测试和文档，不保留旧体验兼容包袱。

## 当前事实

- 数据库已有 `question_categories` 表，且初始化脚本只内置一条“默认试题分类”。
- 后端已有 `GET /api/admin/question-banks/categories`，但只提供分类读取能力。
- 后端题库创建和编辑已经校验 `categoryId` 必须存在。
- 前端 `题库管理` 已是左侧题库树、右侧试题列表的一体化工作台。
- 当前题库树展示分类节点和题库节点，题库节点展示题量；分类节点没有创建、编辑、删除操作。
- 后端 `/api/admin/questions` 仍作为题库管理内部试题 CRUD 接口使用，不作为独立菜单模块。

## 生产级验收保护点

- 分类必须能创建、编辑、刷新读回并出现在题库树一级节点。
- 新建题库必须能选择新建分类，保存后题库出现在该分类下。
- 选择分类后新建题库，默认归属当前分类；选择题库后新建题库，默认归属该题库所在分类。
- 空分类可以删除，删除后树刷新且不残留选中状态。
- 已有题库的分类不能删除，必须返回明确错误，不破坏题库和试题数据。
- 题库和试题原有保存、附件预览、导入导出、题量刷新、考试组卷和发布快照能力不能回退。
- 真实浏览器验收必须覆盖分类创建、分类编辑、分类下新建题库、非空分类删除拦截和空分类删除。

## 目标

- 把试题分类作为题库管理的正式业务对象。
- 增加分类创建、编辑、删除接口和类型化请求。
- 在题库工作台左侧树提供分类操作入口。
- 新建题库逻辑按当前选中的分类或题库推导默认分类。
- 同步后端测试、前端 API 单元测试、Playwright 真实浏览器测试、`spec.md` 和 `README.md`。

## 不做范围

- 不恢复独立“试题管理”页面或菜单。
- 不做多级分类；本次分类是一层，分类下是题库，题库下是试题。
- 不做历史数据迁移兼容；初始化脚本表达当前事实。
- 不改变考试发布快照、作答快照和成绩归档规则。
- 不照抄参考平台视觉，只实现 kaoshi 当前主干需要的清晰层级和可操作性。

## 设计

- 后端：
  - `QuestionCategorySaveRequest` 承载分类名称、说明和排序。
  - `POST /api/admin/question-banks/categories` 创建分类。
  - `PUT /api/admin/question-banks/categories/{id}` 编辑分类。
  - `DELETE /api/admin/question-banks/categories/{id}` 删除空分类。
  - 删除分类前统计 `question_banks.category_id`，非空返回业务冲突。
  - 分类名称使用数据库唯一约束和服务层校验保护。

- 前端：
  - 题库树标题区增加“新建分类”。
  - 分类节点显示题库数量，并提供“新建题库、编辑、删除”操作。
  - 题库节点保留题量、状态和编辑题库能力。
  - 分类表单包含名称、说明和排序。
  - 删除分类使用确认弹窗，失败时展示后端错误。

- 测试：
  - 后端测试覆盖分类创建、编辑、分类下建题库、非空分类删除失败和空分类删除成功。
  - 前端单元测试覆盖 API client 分类接口。
  - Playwright 覆盖真实浏览器分类生命周期。
